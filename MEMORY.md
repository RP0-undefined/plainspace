# Plainspace Memory (profile)

Extends `SKILL.md`. Core rules (token principle, audience, index-first) remain law.
What core lacks for *memory* is a lifecycle. This profile adds it:

**capture → consolidate → recall → forget**

The three memory kinds map onto existing Plainspace layers:
episodic = `log.md` + `inbox/` · semantic = `knowledge/` · procedural = stages/playbooks.

---

## 1. The layer pyramid

Same shape as mem0/Letta/Tencent-style systems, but every layer is a plain file and
each higher layer keeps a path back to the raw evidence below it (§4a). Compression is
never irreversible.

| layer | what | lives in | generated by |
|-------|------|----------|--------------|
| L0 raw | ground-truth captures + change history, immutable | `archive/inbox/`, `log.md` | capture, then archived on consolidation |
| L1 atom | one-fact-per-line observations, unformatted | `inbox/` | capture (§3) |
| L2 concept | consolidated durable facts | `knowledge/` | consolidation (§4) |
| L3 core | the always-loaded distilled summary | `# Core` block in `index.md` | consolidation (§4c) |

Drill-down path (all by frontmatter/link, no grep): `# Core` line → `knowledge/` concept
→ `derived_from` → archived capture → `source` (external origin).

---

## 2. Layout

```
workspace/
├── index.md              # "# Core" block (above the map marker) + generated map
├── log.md                # episodic record, append-only, newest first
├── inbox/                # L1: cheap captures, zero ceremony (§3)
├── knowledge/            # L2: consolidated durable facts — semantic memory
├── playbooks/            # procedural memory (optional)
├── 90_consolidate/       # recurring maintenance stage (§4)
├── archive/              # forgotten facts — excluded from recall (§6)
│   └── inbox/            #   L0: consolidated raw captures, kept for provenance
├── tools/psindex.py      # optional derived index, rung 2 (§5)
└── memory.db             # DISPOSABLE cache — never the truth; gitignore it
```

---

## 3. Added frontmatter fields

| field           | applies to         | meaning |
|-----------------|--------------------|---------|
| `status`        | any                | `current` (default) \| `superseded` \| `archived` \| `consolidated` |
| `supersedes`    | knowledge          | path of the fact this one replaced |
| `superseded_by` | archive            | path of the replacement |
| `derived_from`  | knowledge          | list of archived-capture paths this fact was consolidated from (internal provenance) |
| `promoted_to`   | archive/inbox      | list of knowledge paths a consolidated capture fed into |
| `source`        | any                | external provenance: URL, email id, `"call 2026-07-04"`, … |
| `confidence`    | any                | `high` \| `medium` \| `low` |
| `last_verified` | knowledge          | ISO date the fact was last confirmed true |

`derived_from` (internal, → capture) and `source` (external origin) together give the full
drill-down chain. Producers MAY add keys; consumers preserve unknowns.

---

## 4. Capture — writes must cost nothing

- Mid-task, write observations to `inbox/<date>-<slug>.md`. Frontmatter: `type: Capture`, `source`. Body: one fact per line.
- Do NOT update `index.md`. Do NOT format into a concept. Do NOT deduplicate.
- Rationale: any ceremony at capture time and the agent stops saving. `inbox/` is a buffer, not storage — consolidation empties it.

---

## 5. Consolidate — the stage that makes it a memory

Lives at `90_consolidate/_stage.md`. Triggers are declared as data in its frontmatter
(`triggers: {inbox_files, oldest_days, warmup}`); defaults `10` / `7d`.
**Warmup:** on a young workspace (< ~20 knowledge concepts), consolidate at 2–3 captures
and double the threshold each run until the steady-state trigger — a new memory is most
valuable, and most neglected, at the start.

### 5a. Process (per run)
- **Pre-commit** (if git repo): `git add -A && git commit -m "memory: pre-consolidate"` (local only). `-am` would miss new untracked captures/concepts — always `add -A`.
- **Dedup then promote**: for each inbox fact, search `knowledge/` for the same fact (rung 2 on key terms; rung 3 if available).
  - Duplicate → merge into the existing concept: keep highest `confidence`, union `source`, refresh `last_verified`.
  - New fact → create/extend a concept. Set `derived_from` to the capture path(s); carry `source`, `confidence`.
  - Contradiction → newer wins unless its `confidence` is lower. Loser → `archive/` with `status: superseded` + `superseded_by`. Move history, never rewrite it.
  - Ambiguous → leave in `inbox/`, flag in `log.md` for a human.
- **Archive captures, don't delete** (§4a): each consolidated capture → `archive/inbox/<file>` with `status: consolidated` + `promoted_to: <knowledge path(s)>`. The raw evidence behind every promoted fact survives.
- **Demote**: knowledge unreferenced and `last_verified` > 180 days → `archive/`, `status: archived`.
- **Core refresh** (§5c).
- **Reindex**: regenerate maps (`psindex.py map`); rebuild `memory.db` if used. Append a run summary to `log.md` with counters: promoted / merged / superseded / archived / left-ambiguous (`psindex.py stats`).
- **Post-commit** (if git repo): `git add -A && git commit -m "memory: consolidate <date> — <counters>"`. The diff against pre-consolidate is the human review gate.

### 5b. Traceability (the law consolidation must not break)
Forgetting is *move to archive*, never delete — for captures too. A promoted `knowledge/`
fact MUST carry `derived_from`; the archived capture MUST carry `promoted_to`. Acceptance:
from any concept an agent reaches the original raw capture by following frontmatter paths
only. `psindex.py stats` reports knowledge facts lacking provenance as a hygiene counter.

### 5c. The `# Core` block has a lifecycle
`# Core` is L3: the always-loaded distilled memory. Left hand-written it silently rots, so
consolidation owns it:
- Regenerate/review Core every ~50 promotions (configurable) or whenever a concept it cites is superseded.
- **Every Core line MUST link to the concept it distills** (drill-down, §1).
- **Budget: ≤ ~15 lines / one screen.** Core is the only always-loaded file — it must be the most token-disciplined thing in the workspace. Over budget → push detail down into `knowledge/`, keep only the pointer in Core.

### 5d. Boundary — one writer
One workspace = one writing agent at a time. This profile has no locking; concurrent
writers corrupt consolidation. Multiple agents need separate workspaces, or all writes
serialized through the one agent that runs consolidation. Consolidation MAY run as a
dedicated subagent (its `_stage.md` is the complete prompt); capture and targeted recall
stay with the calling agent.

Profile extension to core §6: **maintenance stages (numbered 90+) MAY write across the
workspace** instead of only `output/`. Numbering convention: 90+ = recurring maintenance.

---

## 6. Recall — the 3-rung ladder

Protocol is fixed; the substrate scales underneath it. Climb one rung only when the rung below fails.

| rung | mechanism                                        | good until    | added tooling |
|------|--------------------------------------------------|---------------|---------------|
| 1    | `index.md` maps (per-directory, generated)       | ~500 concepts | none |
| 2    | grep, or FTS: `python3 tools/psindex.py search "query"` | ~50k concepts | 1 stdlib script |
| 3    | hybrid BM25 + embeddings behind the same verb (`tools/pssearch3.py`) | beyond | optional embeddings |

**Budget (never let memory block the task).** Retrieve at most ~5 candidates
(`--limit`, default 5), then open the ONE file you need and stop — recall costs one read,
not a fan-out. If search tooling is missing, slow, or errors, fall back one rung and
continue; a degraded recall must never stall the work. `--since YYYY-MM-DD` narrows to
recently-updated facts.

**Law: files are the truth; every index is derived and disposable.**
`memory.db` can always be rebuilt (`psindex.py build`), and auto-rebuilds when a file is
newer than it or the count changed. If db and files disagree, files win.

---

## 7. Forget

- Default recall scope excludes `archive/` and any `status` in {`superseded`, `archived`, `consolidated`}.
- Archiving is a **move, not a delete** — for both concepts and consumed captures. Provenance stays greppable, supersession and `derived_from` chains stay walkable.
- Recall quality depends on the size of the **active set**, not total size. Consolidation + archive keep the active set flat while total grows.

---

## 8. First adoption

On first use of a workspace, read `SETUP.md` once (if present): it lets you make the
capture checkpoint deterministic and schedule consolidation by configuring your own
harness — with the human's approval. Never reload it afterwards.

---

## 9. Reading protocol (agent)

1. Read `index.md`. Load the `# Core` block (above the map marker) — it is the always-in-context memory; each line links to the concept behind it.
2. Resolve the target from maps (rung 1); else search (rung 2/3), ≤5 candidates. Open ONE file. Stop.
3. Trust frontmatter. Check `status` before acting on a fact. Follow `supersedes`/`derived_from` chains only when auditing.

# Notes

Why an inbox: agents skip saving when saving is expensive. The single biggest failure
mode of file-based memory is an empty one. Capture is deliberately allowed to be messy;
consolidation pays the formatting cost later, in batch.

Why archive instead of delete — even for raw captures: a memory that deletes its evidence
cannot answer "why did we think X?". Keeping consumed captures in `archive/inbox/` with
`promoted_to` preserves an unbroken L3→L0 drill-down at zero hot-path cost (recall excludes
archive by default).

Why Core has a budget: it is loaded on every turn. An unbudgeted always-loaded file is how
a memory system quietly reintroduces the token cost the whole format exists to remove.

Why the db is a cache: the moment the database becomes the truth, the workspace stops
being reviewable with `cat` and shippable with `git clone` — the two properties the whole
format exists to protect. Deriving it from frontmatter keeps both worlds.
